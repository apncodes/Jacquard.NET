⏺ Good question — there are real implications worth thinking through now before
  you publish anything.

  NuGet Package Ownership

  NuGet package IDs are first-come, first-served and permanent. Once you push
  Strands.Core under your personal NuGet account, that account owns it.
  Transferring ownership to an org account later requires:
  - The original owner to add the new owner via NuGet.org UI
  - Then optionally remove themselves

  This works fine IF the move is just to an org account you control. But if
  AWS/Strands team takes over the repo, they'd need you to transfer ownership on
   NuGet.org separately — NuGet has no concept of "follows the repo".

  ---
  Naming Risk

  The bigger question: does the Strands team have a view on the Strands.* 
  namespace?

  If they eventually want to publish official Strands.* packages from their own
  NuGet account, your packages would conflict — NuGet IDs are globally unique.
  Options:

  ┌───────────────────────────────┬─────────────────────────────────────────┐
  │           Scenario            │             Recommendation              │
  ├───────────────────────────────┼─────────────────────────────────────────┤
  │ You'll be the official        │ Publish now under your account,         │
  │ maintainer under the Strands  │ transfer later                          │
  │ org                           │                                         │
  ├───────────────────────────────┼─────────────────────────────────────────┤
  │ AWS/Strands team will own it  │ Wait — let them create the NuGet org    │
  │ long-term                     │ account first, publish under that       │
  ├───────────────────────────────┼─────────────────────────────────────────┤
  │ Uncertain                     │ Use a scoped name like StrandsDotNet.*  │
  │                               │ or Strands.Net.* as a holding name      │
  └───────────────────────────────┴─────────────────────────────────────────┘

  ---
  Repo Move Implications

  The repo URL is baked into Directory.Build.props:
  <PackageProjectUrl>https://github.com/apncodes/strands.net</PackageProjectUrl>
  <RepositoryUrl>https://github.com/apncodes/strands.net</RepositoryUrl>

  Once moved, any already-published packages will have stale URLs. Not a
  breaking change, but looks unprofessional. The fix is a version bump +
  re-publish after the move.

  ---
  My Recommendation

  Don't publish to NuGet.org yet. Instead:

  1. Align with the Strands team on whether Strands.* is the right namespace for
   a community/official package
  2. Get the repo moved first (GitHub redirects from old URL automatically)
  3. Update RepositoryUrl in Directory.Build.props
  4. Publish v0.1.0-preview.1 from the canonical location under the right NuGet
  account

  This avoids any ownership awkwardness and means the first public package
  points to the right home.

❯ Week 7 — Strands.AgentCore: Hosting Wrapper + Managed Services                
You are adding Amazon Bedrock AgentCore support to the Strands.NET SDK.         
Run this prompt from the repo root directory.                                   
The correct mental model — read this before writing a single line               
AgentCore is NOT a model provider. AgentCore is NOT an agent type.              
AgentCore is a HOSTING PLATFORM and a set of MANAGED SERVICES.                  
The Python Strands SDK makes this crystal clear:                                
pythonfrom bedrock_agentcore.runtime import BedrockAgentCoreApp                 
from strands import Agent                                                       
                                                                                
app = BedrockAgentCoreApp()     # hosting wrapper                               
agent = Agent()                  # completely unchanged agent                   
                                                                                
@app.entrypoint                                                                 
def invoke(payload):                                                            
    result = agent(payload.get("prompt"))                                       
    return {"result": result.message}                                           
                                                                                
app.run()                        # starts /invocations HTTP server              
The agent keeps its own model (Bedrock, OpenAI, Ollama — anything).             
The agent keeps its own tools, hooks, sessions — everything.                    
BedrockAgentCoreApp wraps the agent as an HTTP service that AgentCore           
Runtime knows how to host, scale, and isolate.                                  
The .NET equivalent must follow exactly the same pattern:                       
                                                                                
The hosting wrapper exposes the agent via HTTP endpoints                        
The agent code is completely unchanged                                          
AgentCore managed services (Memory, Browser, Code Interpreter)                  
are optional ITool implementations the agent can choose to use                  
                                                                                
                                                                                
Critical architectural principle                                                
AgentCore is ADDITIVE and OPTIONAL. Nothing built in Weeks 1-6 changes.         
                                                                                
Strands.Core — zero changes                                                     
Strands.Models.Bedrock — zero changes                                           
Strands.Tools — zero changes                                                    
Strands.MultiAgent — zero changes                                               
Strands.Extensions.DependencyInjection — zero changes                           
All existing tests — must still pass unchanged                                  
                                                                                
This package adds two things:                                                   
                                                                                
A hosting wrapper (MapAgentCoreEndpoints) that makes any Strands.NET            
agent deployable to AgentCore Runtime via HTTP protocol compliance              
Optional managed service tools (Memory, Browser, Code Interpreter)              
that agents can choose to use as ITool implementations

TASK 3 — /invocations endpoint handler                                          
File: src/Strands.AgentCore/Hosting/AgentCoreInvocationHandler.cs               
This is the core of the hosting wrapper. Receives POST /invocations,            
invokes the completely unchanged Strands.NET agent, returns response.           
Supports both non-streaming (JSON) and streaming (SSE) modes.                   
The agent (IAgent) is resolved from DI. The handler has no knowledge            
of what model the agent uses, what tools it has, or how it works                
internally. It just calls InvokeAsync or StreamAsync.                           
Streaming detection: check Accept header for text/event-stream.                 
AgentCore Runtime sends this header when it wants token streaming.              
For SSE streaming, write each TextDeltaEvent as:                                
data: {"text": "token"}\n\n                                                     
Terminate with:                                                                 
data: [DONE]\n\n                                                                
Flush after each write — AgentCore Runtime buffers until flush.
